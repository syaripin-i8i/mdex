## 背景

2026-04-22 時点で CI は green に復帰済み。

確認済み run:
- https://github.com/syaripin-i8i/mdex/actions/runs/24740055135

直近で解消した問題:
- `install_from_pylock.py` が clean runner 上で `packaging` 未導入のため bootstrap 失敗
- `tests/test_install_from_pylock_script.py` が `py3.10` で `tomllib` 直 import により collection error

現在の状態:
- `install_from_pylock.py` は `packaging` を優先しつつ、bootstrap 用 safety net として `pip._vendor.packaging` に fallback している
- pytest 側では `tests/conftest.py` で `py3.10` 用に `tomli -> tomllib` shim を注入している
- `tests/test_install_from_pylock_script.py` には bootstrap import と hash catalog closure の回帰テストが入っている

DeepResearch 監査で出た論点:
- `pip._vendor.packaging` は pip 内部 API 依存で、長期的な安定性は低い
- test collection を救うための `tomllib` shim は現状合理的だが、将来的により局所的な代替があるか再評価余地がある
- CI matrix と hash completeness の検査は既に一定量あるが、追加の verify job や古い pip matrix が必要かは方針整理の余地がある

ただし **現時点では CI green であり、即修正必須の不具合ではない**。そのため今回は「今の guardrail を壊さず、長期的に bootstrap 依存境界をどう整理するか」を follow-up として残す。

## 目的

1. `pip._vendor.packaging` fallback を長期的に維持するか、別方式へ移行するかを判断する
2. bootstrap script が lockfile 依存自身に先行依存しない、という制約を明文化した上で実装を整理する
3. `py3.10` 互換と CI green を維持したまま、必要な追加ガードだけを導入する

## 対象

## 論点

### A. `pip._vendor.packaging` の扱い
- 現状は bootstrap 失敗を防ぐための pragmatic fallback として妥当
- private API 依存のため、恒久解としてよいかは別途判断が必要
- 候補: 現状維持 + rationale 明文化 / bootstrap path で `packaging` 依存自体を減らす / CI/tooling 専用依存として整理

### B. `tomllib` shim の扱い
- `tests/conftest.py` の shim は `py3.10` collection error を防ぐ目的に有効
- native `tomllib` を上書きしない `setdefault` 実装なので危険度は低い
- 将来 test module 構成が変わる場合、より局所的な互換層へ寄せる余地があるか再確認

### C. CI / verify job
- 既存: bootstrap import fallback / pylock coverage / CI matrix closure
- 追加 verify job や古い pip matrix は support policy とメンテ負荷を見て判断

## 完了条件

2. その判断に応じて、必要なコード / テスト / docs だけが最小変更で入っている
