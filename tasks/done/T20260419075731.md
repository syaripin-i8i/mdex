## 背景
3 OS × 3 Python の CI matrix 拡張後、Windows 3 jobs が全て失敗。`tests/test_cli_e2e.py::test_quality_repo_scan_first_related_enrich_e2e` で `python -m runtime.cli enrich ...` を subprocess 呼び出しした際、Windows 上で piped stdout がロケール依存エンコーディング (cp1252/cp932) にフォールバックし、非 ASCII を含む出力で `UnicodeEncodeError: 'charmap' codec can't encode characters in position 179-188` が起きる。CI 回避ではなく**本物のユーザーバグ**として根本修正する。

## 対象
- runtime/cli.py: エントリポイントで `sys.stdout` / `sys.stderr` を UTF-8 に再設定する。`sys.stdout.reconfigure(encoding="utf-8", errors="strict")` を main() 先頭（argparse 以前、ただし import 直後）で呼ぶ方針。Python 3.10+ 前提なので reconfigure で十分。再設定が困難な環境（stdout が tty でない／既に buffer 取り出し済み等）には try/except で safely fallback し、失敗しても CLI は続行
- runtime/ 配下で `json.dump(..., ensure_ascii=False)` を使っているものは維持（人が読むため）。`ensure_ascii=True` へ退避するような逃げは取らない
- 既存の subprocess 実行（enrich のインターナルで child を spawn している場合）も環境変数 `PYTHONIOENCODING=utf-8` を明示するか、あるいは親が UTF-8 を固定していれば不要。念のため `subprocess.run` 呼び出し側で `encoding="utf-8"` を揃える
- tests/test_cli_e2e.py などで `subprocess.run(..., capture_output=True)` している箇所は `encoding="utf-8"` を追加（デコードエラーを抑える防御）

## 完了条件
1. 3 OS × 3 Python の 9 CI jobs が全て green（windows-latest 3 個を含む）
2. ローカル (ubuntu/macos/windows) いずれでも `python -m runtime.cli enrich` で日本語を含む出力が `UnicodeEncodeError` を起こさない
3. 既存 72 passed を維持（regression なし）
4. 修正は UTF-8 固定化のみ。出力契約・エスケープ規則は変更しない

## 非目標

- ロケール検出や自動切替
- Python 3.9 以下のサポート
- 出力 JSON の ensure_ascii 化（回避策としての退避は禁止）
